# ProsperPals Batch 01 Interview Command Center

Use this file as the **daily operating surface** for the first three ProsperPals interviews.

This artifact supports the current BMAD step:

**`alpha-readiness / interview-evidence-and-hosted-hardening-execution`**

It does **not** soften the hosted-alpha **NO-GO**.

It exists because the repo already had the right strategy artifacts:
- a 12-slot cohort design,
- a recruiting/outreach pack,
- a private candidate-registry contract,
- a Batch 01 assignment board,
- a canonical evidence tracker,
- and session/synthesis templates.

What it still lacked was one compact command surface for the **first real operating week**.

Without that, the likely failure mode was predictable:
- leads sit across multiple private threads,
- no one knows whether `PP-B01`, `PP-I01`, and `PP-M01` each have an active owner,
- founder-network replies arrive first and quietly dominate,
- scheduling gets ahead of evidence hygiene,
- and Batch 01 looks active while still not meeting its credibility bar.

This command center turns Batch 01 into a bounded, reviewable operating lane instead of a loose founder-memory task.

## 1. Elicitation methods applied

### First Principles
Used to reduce Batch 01 to the minimum real management question: are the three required contrastive slots actually moving toward trustworthy sessions, or are they only producing activity theater?

### Pre-mortem
Used to design against the most likely Batch 01 failure patterns: founder-network overfill, inactive backups, no non-founder-network recruit, stale replied leads, and scheduled calls that are not actually note-ready.

### Cross-Functional War Room
Used to align recruiting ownership, design-learning goals, founder scheduling reality, and evidence-quality rules into one daily check surface instead of scattered docs.

## 2. What this file is for

Use this file to answer five questions every day while Batch 01 is open:

1. Do all three slots have an active owner?
2. Does each slot have at least one backup candidate?
3. Is at least one non-founder-network recruit truly active?
4. Are any candidates about to be scheduled without clean alias handoff?
5. Are we about to run calls that will fail the same-day evidence bar?

If this file is stale, Batch 01 is not under control.

---

## 3. Batch 01 success definition

Batch 01 is healthy only when all of the following are true:

- `PP-B01`, `PP-I01`, and `PP-M01` each have one active candidate owner
- each slot has at least one viable backup candidate offline
- at least one active slot is **not** founder-network sourced
- all three slots have a one-sentence fit rationale
- no slot is relying on a `screened-borderline` candidate as the only option
- no session is marked scheduled until alias handoff into repo artifacts is complete
- deterministic note files are ready before the first call
- same-day logging discipline is still realistic for all booked times

Anything less is motion, not readiness.

- `Ready to schedule?` can only be `yes` when the private-registry handoff checklist is satisfied.

## 8. Daily operating ritual

### Start-of-day check
- review the three slot-health rows
- review funnel depth by slot
- review source-mix panel
- identify the one biggest blocker for each slot
- assign one next action owner per slot

### End-of-day closeout
- update funnel counts honestly
- note any stale `replied` or `screened-fit` leads older than 48h
- record whether any slot is now schedule-safe
- record whether Batch 01 can still hit the non-founder-network bar

If this ritual is skipped for 2+ days, Batch 01 should be treated as stale.

---

## 9. Escalation triggers

Pause scheduling or re-open sourcing if any of the following become true:

- 2 of 3 active slots are founder-network sourced
- any slot has `backup count = 0` for more than 48 hours
- a candidate is about to be scheduled while alias handoff is incomplete
- a `screened-borderline` candidate becomes the only option for a slot
- the first booked call has no realistic same-day note window
- a single social cluster dominates more than one active slot

These are not small admin misses. They directly lower evidence quality.

## 10. Batch 01 meeting script

Use this 5-minute script in founder/design check-ins:

1. **Slot review:** what is the active owner + blocker for `PP-B01`, `PP-I01`, and `PP-M01`?
2. **Source review:** do we still have at least one real non-founder-network path alive?
3. **Backup review:** which slot is one ghosting event away from stalling?
4. **Scheduling review:** which slots are truly handoff-complete vs only socially convenient?
5. **Evidence review:** if a session happened tomorrow, would the note and threshold chain still hold?

If the team cannot answer these five in under 5 minutes, Batch 01 is under-instrumented.
