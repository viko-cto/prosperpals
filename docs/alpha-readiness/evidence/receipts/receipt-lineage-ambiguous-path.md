# Receipt Lineage — Ambiguous / Low-Confidence Path

**Date:** 2026-03-22  
**Evidence type:** repo-state execution note from current code  
**Decision use:** proves the current demo slice flags low-confidence receipt parses visibly and does not auto-post ambiguous output.

## Verdict

The implemented receipt flow does handle ambiguity honestly at the UI/state level:
- low-confidence parses are labeled as such,
- review status stays `needs_review`,
- the user is told the parse is too fuzzy to trust,
- and nothing becomes canonical truth until explicit confirmation.

### Relevant behavior
- `getConfidenceScore(...)` can leave a candidate below the review threshold.
- `getConfidenceLabel(...)` maps that state to **“Low-confidence parse — confirmation required.”**
- `getReviewStatus(...)` keeps the record in `needs_review` unless the parse is strong enough to be `ready_to_confirm`.
- The page shows the candidate’s confidence and review state before any posting step is available.
- The explainability layer marks a pending candidate as `review-required` rather than pretending it is already trustworthy.

## Trace

### 1. Candidate enters as ambiguous
The receipt runtime stores the candidate with a low confidence score and a review message that explicitly says the parse is too fuzzy to trust.

### 2. UI keeps uncertainty visible
`/app/receipts` surfaces:
- the confidence label,
- the review message,
- the current state (`needs review`),
- and the posting rule that canonical truth waits for confirmation.

### 3. Explainability repeats the same trust posture
`getDemoExplainabilityCards(...)` turns a pending candidate into a `review-required` explanation card instead of treating it as a finished money event.
